## 5. 重要なコード・設計の方針

### アーキテクチャ概要

```
Frontend (Vue/Vite) → FastAPI backend → Postgres (pgvector) + Redis
                    ↘ grading-worker (arq)
                    ↘ curriculum embeddings (HuggingFace / stub)
                    ↘ upload storage (local disk or S3)
```

### E2E テスト設計（共有 DB 問題）

CI では **単一 Postgres** を全 E2E が共有する。admin テストがカリキュラムを変更するため:

- **実行順序:** `00-` プレフィックスで dashboard を先に実行
- **クリーンアップ:** `resetAiDrivenDevCurriculumBaseline()` で Phase 1 task 4 / Phase 5 を削除
- **backend 要件:** `CLAUDE_STUB_MODE=true`（採点スコア `stub:weak`→55, `stub:ok`→75, `stub:great`→92）

### ファイルストレージ（Sprint 27）

| 設定 | 説明 |
|------|------|
| `UPLOAD_STORAGE_BACKEND=local` | デフォルト。`upload_dir` + Compose volume |
| `UPLOAD_STORAGE_BACKEND=s3` | multi-replica 本番向け。DB に `s3://bucket/key` を保存 |

grading-worker も同じ env を参照するため、backend replica 間でファイル共有可能。

### Embeddings ライフサイクル（Sprint 20〜25）

- publish / add_task / move_task / delete_task / delete_phase 時に embeddings を整合
- CI では `EMBEDDING_STUB_MODE=true` で HuggingFace 429 を回避
- 新規 course 作成時は API 経由で自動 seed。初回本番は `scripts/seed_embeddings.py`

### カリキュラムデータ

- `ai-driven-dev`: 4 phases × 3 tasks = **12 tasks**（ベースライン）
- seed は migration `20260613_53858e23cd1b_sprint9_curriculum_editing.py` に凍結
- admin E2E が一時的に task 4 / phase 5 を追加するが、テスト後に削除する設計

### 本番デプロイパス

| パス | 用途 |
|------|------|
| `make prod` | bundled Postgres/Redis |
| `make prod-tls` | + Caddy HTTPS |
| `make prod-managed` | 外部 DATABASE_URL / REDIS_URL |
| ALB + Terraform | `docs/infra/alb-deploy.md` + `infra/terraform/alb/` |

### DB 不整合時の復旧

pytest 実行後に dev DB を共有していると `alembic_version` のみでテーブルが空になることがある:

```bash
docker compose exec -T postgres psql -U postgres -d ai_tutor \
  -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;"
cd backend && uv run alembic upgrade head
```

## 8. セッション再開時のクイックスタート

```bash

# インフラ起動
docker compose up -d postgres redis

# migration

# backend（ローカル）
```
